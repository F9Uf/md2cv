<objective>
The browser-print PDF export emits N+1 physical pages (an extra trailing blank page after the content). Add a single CSS override inside the existing `@media print` block in `src/styles/pages.css` so the last `.pagedjs_page` does NOT force a physical page break after itself.

Purpose: Fix the trailing blank page on every PDF export without altering screen preview, multi-page handling, margins, or any of the three templates.
Output: Modified `src/styles/pages.css` (one new rule, ~4 lines).
</objective>

<diagnosis>
**Confirmed root cause — Hypothesis 3.**

Paged.js's polyfill stylesheet (`node_modules/pagedjs/dist/paged.esm.js` lines 27423–27431) injects this rule at runtime:

```css
@media print {
  .pagedjs_page {
    margin: 0;
    padding: 0;
    max-height: 100%;
    min-height: 100%;
    height: 100% !important;
    page-break-after: always;   /* ← every .pagedjs_page, including the last */
    break-after: page;          /* ← every .pagedjs_page, including the last */
  }
}
```

`break-after: page` / `page-break-after: always` on every `.pagedjs_page` is intentional — it forces the browser print engine to chunk each paged.js page to its own physical sheet (otherwise multi-page resumes would collapse onto one sheet). But on the LAST `.pagedjs_page` the same rule forces a break *after* it, producing a trailing empty physical page.

Evidence ruling out the other hypotheses:
- **H1 / H2** (polyfill `@page` size, `.pagedjs_pages` flex): the polyfill `@page` block (line 27205+) sets sizes that are consistent with our 0-gutter `@page`, and the flex container has no print-time height contribution beyond its sole child. These would produce sizing/clipping bugs, not an exact +1 page.
- **H4** (trailing pending node): paged.js cleans up before resolving `previewer.preview(...)`; the awaited flow we use in `usePagedjsPreview.ts:80-84` has no `.pagedjs_page_pending` left in the DOM.
- **H5** (`width: 210mm` cloak): print rule in `index.css:22` sets `width: 100%`, which has equal specificity and wins by source order. Not the cause.

**Fix:** Inside the existing `@media print` block in `src/styles/pages.css`, override `break-after` and `page-break-after` to `avoid` on `.pagedjs_page:last-child`. This neutralizes the trailing break for single-page AND multi-page resumes (the last page is always the only one that should not force a successor sheet). The rule is specifically scoped to the last page so multi-page pagination is unaffected.

**File choice:** `src/styles/pages.css` already owns the print-scope reset for `.pagedjs_page` (lines 33-38) — adding the new rule there keeps cohesion with neighboring print overrides. `index.css` stays the source of truth for app-shell hiding and physical `@page`.
</diagnosis>

Inside the EXISTING `@media print { ... }` block (currently lines 33-50) in `src/styles/pages.css`, immediately after the existing `.pagedjs_page { box-shadow: none; border: none; margin-bottom: 0; }` rule and BEFORE the `.pagedjs-scale-wrapper` rule, add the following new rule:

```css
  /* Paged.js's polyfill injects `break-after: page; page-break-after: always`
   * on EVERY .pagedjs_page in print context (paged.esm.js lines 27423-27431).
   * That's correct for multi-page resumes (it chunks each .pagedjs_page to its
   * own physical sheet), but on the LAST page it forces an extra blank
   * physical page after the content. Override the trailing break — scoped to
   * :last-child only so multi-page pagination is preserved. */
  .pagedjs_page:last-child {
    break-after: avoid !important;
    page-break-after: avoid !important;
  }
```

Constraints:
- Do NOT touch any other file. Do NOT touch `src/index.css`, `src/App.tsx`, `src/components/PrintMount.tsx`, or `src/hooks/usePagedjsPreview.ts`.
- Do NOT remove, paraphrase, or move the existing comments in the file. The Phase 7 header comment, the Phase 9 ZOOM-01 comment, and the "Belt-and-suspenders" comment at lines 29-32 must remain verbatim.
- Use `!important` because paged.js's injected `break-after: page` is itself `!important`-level via the polyfill's internal stylesheet ordering — without it, the override may lose specificity battles in some browsers.
- Use BOTH the modern `break-after` and the legacy `page-break-after` for cross-browser coverage (Chrome/Edge accept either; Safari historically prefers the legacy form).
- Use `:last-child` (not `:last-of-type`) because `.pagedjs_pages` may contain non-`.pagedjs_page` siblings in some paged.js versions; `:last-child` correctly targets only the case when the .pagedjs_page is in fact the final child.
