# Quick Task 260521-m56 — Summary

## Problem

Typing in the editor caused the paged.js preview to visibly **blink** on every debounced reflow. The user correctly attributed it to the zoom-to-fit-the-page logic added in phase 09.

## Root cause

`src/components/Preview.tsx`, paged.js reflow `useEffect`: every reflow called `setPageCount(null)`. The render gate `zoomReady = pageCount !== null` then dropped `zoomStyle` from the wrapper, removing the current fit-zoom (e.g. `0.6`). Content jumped to 100% → snapped back when paged.js finished and `setPageCount(count)` re-ran.

## Fix

Removed the `setPageCount(null)` reset. The previous count remains in state during reflow, so `zoomReady` stays `true` and `zoomStyle` keeps `zoom: <scale>` applied throughout. The pill label briefly shows the stale count until the new render resolves (~150ms) — acceptable trade vs. the visible scale snap.

## Considered and rejected — off-screen staging + atomic swap

An initial attempt rendered paged.js into a hidden staging `<div>` and swapped via `root.replaceChildren(...)` to also kill the brief paged.js-render blank gap. It crashed in paged.js:

```
TypeError: Cannot read properties of null (reading 'getAttribute')
  at findElement (pagedjs.js)
  at Page.checkUnderflowAfterResize  ← scheduled in requestAnimationFrame
```

Each `Page` installs a `ResizeObserver` (`paged.esm.js:2413`). Moving rendered pages to a new parent triggers a rAF that walks the now-consumed source `contents`, hitting `findElement → null`. Disabling listeners pre-swap or destroying the chunker pre-swap leaks DOM in awkward ways — approach abandoned. The remaining brief render gap inside the stable zoom wrapper is acceptable.

## Files changed

- `src/components/Preview.tsx` — removed `setPageCount(null)` from line 50; updated explanatory comment.
