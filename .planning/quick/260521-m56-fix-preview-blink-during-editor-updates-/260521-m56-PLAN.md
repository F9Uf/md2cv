# Quick Task 260521-m56: Fix Preview Blink During Editor Updates

## Problem

When the user types in the editor, the debounced HTML update triggers a paged.js reflow in `src/components/Preview.tsx`. The preview visibly **blinks** on every keystroke (after the 150ms debounce settles). The user correctly suspected it relates to the "zoom-to-fit-the-page" logic added in phase 09.

## Root cause (two compounding effects)

Inside the first `useEffect` of `Preview.tsx` (the paged.js reflow effect):

1. **Zoom-snap blink** — line 50 calls `setPageCount(null)`. The render gates zoom on `pageCount !== null` (line 163 `zoomReady = pageCount !== null`, line 164 builds `zoomStyle`). When `pageCount` resets to `null`, `zoomStyle` becomes `undefined` → the current fit-zoom (e.g. `0.6`) is **removed** during reflow → content jumps to 100% → snaps back to the fitted scale only after paged.js finishes and re-sets `pageCount`.
2. **Blank-frame blink** — line 48 calls `root.innerHTML = ''` synchronously, wiping the previously rendered pages from the DOM. paged.js then mounts new pages incrementally over the next ~10ms + render time, leaving a visible blank gap.

The `pageCount !== null` gate was added in commit `e9a3226` to defer initial zoom until paged.js measured the A4 container. It is only needed on the **first** render — on subsequent renders the wrapper already has stable layout, so resetting `pageCount` to `null` on every edit is what causes the visible scale snap.

## Fix

Single file change in `src/components/Preview.tsx`, paged.js reflow `useEffect`:

### Change — keep zoom applied across reflows

Remove `setPageCount(null)` (line 50). Leave the previous count in state so `zoomReady` stays `true` and `zoomStyle` keeps the current `zoom: <scale>` applied throughout the reflow. The pill label will briefly show the stale count (e.g. "Page 2 of 2" for ~150ms while the new render computes) — acceptable trade for eliminating the visible scale snap.

### Considered and rejected — atomic-swap via off-screen staging

Initial attempt rendered paged.js into a hidden staging `<div>` and swapped via `root.replaceChildren(...)`. It crashed in paged.js with:

```
TypeError: Cannot read properties of null (reading 'getAttribute')
  at findElement (pagedjs.js)
  at Layout.findEndToken
  at Page.checkUnderflowAfterResize  ← scheduled in requestAnimationFrame
```

Root cause: each `Page` instance installs a `ResizeObserver` (paged.esm.js:2413) that triggers `checkUnderflowAfterResize` in `rAF`. Moving the rendered pages into a different parent causes the observer to fire under a new size; the rAF then walks the source `contents` (which paged.js has already consumed) and crashes when `findElement` returns `null`. Disabling listeners pre-swap or destroying the chunker pre-swap both leak DOM in awkward ways. Approach abandoned — too risky vs. the small remaining blank-frame gap.
