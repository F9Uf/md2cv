# Phase 7 — UI Design Contract

> Visual and interaction contract for the on-screen A4 page chrome, multi-page auto-flow, and live "Page X of N" indicator. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase introduces no new interactive widgets. It changes how the existing preview pane renders: from a continuous `<div>` to one or more A4 page rectangles with a passive page-count readout. All chrome (header, split-pane, mobile tabs, buttons) is unchanged.

| Property | Value |

## Spacing Scale

Declared values (all multiples of 4 — match existing project usage):

| Token | Value | Usage in Phase 7 |
|-------|-------|------------------|
| xs | 4px | Page indicator inner padding (vertical) |
| sm | 8px | Page indicator inner padding (horizontal); page indicator gap from page bottom |
| md | 16px | Default gap between stacked A4 page rectangles (vertical); pane padding around page stack |
| lg | 24px | Top/bottom padding of the scroll viewport surrounding the page stack |
| xl | 32px | (reserved — not used this phase) |
| 2xl | 48px | (reserved — not used this phase) |
| 3xl | 64px | (reserved — not used this phase) |

**Page-interior margin (printed area):** 15mm on all four sides (locked by CONTEXT.md D-04). This is a physical/print measurement applied via `@page { margin: 15mm }` in paged.js — NOT part of the 4px-multiples on-screen spacing scale.

**Page rectangle physical dimensions:** 210mm × 297mm (A4 portrait). At 96dpi this is approximately 793.7px × 1122.5px. This is dictated by CSS Paged Media and is not negotiable.

Exceptions:
- The 15mm interior page margin is in millimeters (print unit), intentionally outside the 4px on-screen scale.
- A4 page dimensions in mm are outside the 4px on-screen scale (they are physical paper dimensions).

## Typography

Page chrome introduces exactly **one** new piece of UI text — the "Page X of N" indicator. All other text inside page rectangles is governed by the existing per-template `theme-*.css` rules (already locked Phase 6) and not re-specified here.

| Role | Size | Weight | Line Height | Where it appears |
|------|------|--------|-------------|------------------|
| Page counter | 12px (Tailwind `text-xs`) | 500 (Tailwind `font-medium`) | 1.2 (Tailwind `leading-tight`) | Inside preview pane, bottom-right corner of pane, sticky overlay |
| (in-page content) | per-template (`theme-{classic,modern,minimal}.css`) | per-template | per-template | Inside each A4 page box — unchanged from Phase 6 |

Page counter font family: system stack (inherits from app shell). Color: see Color section.

**Why so small:** The counter is a passive readout, not a control. It must not compete with resume content for attention. `text-xs` matches the existing `Header.tsx` button label size and the `theme-classic` `p`/`li` rules — visually consistent with the rest of the app shell.

## Color

The preview-pane surround changes from white (current `bg-white` in `SplitPane.tsx`) to light gray, so the white A4 page rectangles read as discrete sheets. The pane's own background is updated; the split-pane component shell is unchanged.

| Role | Value | Tailwind class | Usage |
|------|-------|----------------|-------|
| Dominant (60%) — page surround | `#f3f4f6` | `bg-gray-100` | Preview-pane viewport background (the area around and between page rectangles) |
| Secondary (30%) — page sheet | `#ffffff` | `bg-white` | The A4 page rectangle itself; user content renders on this surface |
| Accent (10%) — page counter pill | `rgba(17, 24, 39, 0.85)` | `bg-gray-900/85` | Background of the floating "Page X of N" pill |
| Page counter text | `#ffffff` | `text-white` | Foreground of the floating "Page X of N" pill |
| Page rectangle border | `#e5e7eb` (1px) | `border border-gray-200` | Subtle 1px outline on each A4 sheet (under the shadow) so the sheet edge is crisp on the gray surround |
| Page rectangle shadow | `0 2px 8px rgba(0, 0, 0, 0.08), 0 1px 2px rgba(0, 0, 0, 0.06)` | custom (in `index.css`) | Subtle elevation — Word/Docs-style drop shadow |
| Destructive | not applicable | — | No destructive actions in this phase |

Accent reserved for:
- The "Page X of N" floating pill background ONLY.
- Not used on page rectangles, separator chrome, or any other element introduced in this phase.

**60/30/10 rationale:**
- 60% dominant = `bg-gray-100` covers the preview pane background, which is now the largest visible area when pages are stacked.
- 30% secondary = white A4 sheets are the discrete document surfaces.
- 10% accent = the single high-contrast counter pill draws the eye exactly where the user needs to glance to verify page count.

**Why not use the existing blue (`bg-blue-600`):** Blue is already the project's primary-CTA accent (Export PDF button in `Header.tsx`). Reusing it on the page counter would dilute that signal. A neutral dark pill keeps the counter informational, not actionable.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Page indicator (multi-page) | `Page {X} of {N}` (e.g., `Page 1 of 2`) |
| Page indicator (single page) | `Page 1 of 1` — **always visible**, even when there is only one page (so the user always knows the indicator exists and what it shows) |
| Page indicator during initial reflow | `Page – of –` (en-dash placeholder; shown only on first paint before paged.js has emitted a count — typically < 150ms) |
| Empty preview state | **unchanged** — `Start typing markdown to see your resume preview` (existing copy in `Preview.tsx`). When `htmlContent` is empty, NO page rectangle is shown — fall back to the existing centered placeholder text on the gray surround. |
| Error state | not applicable — paged.js failures are silent fallbacks (see Failure Behavior below); no user-facing error message in this phase |
| Destructive confirmation | not applicable — no destructive actions in this phase |
| Primary CTA | not applicable — this phase adds no new CTA. Existing CTAs in `Header.tsx` (Export PDF, etc.) are unchanged. |

**Copy style:** sentence case, no period (matches existing app voice in `Header.tsx` labels: "Import MD", "Download MD", "Export PDF").

**Why always show "Page 1 of 1":** Per CONTEXT.md the requirement (PREV-03) only says the indicator must be visible and update live. Always-showing avoids a layout flash when the user crosses the single-page boundary and removes the "where did the counter go?" cognitive load. The pill is small enough that always-on is unobtrusive.

## Layout & Interaction Specifics

### Page rectangle layout (inside preview pane)

```
Preview pane (was: bg-white, now: bg-gray-100)
│
├── Scroll viewport (overflow-auto, py-6 px-4 — 24px top/bottom, 16px left/right)
│   │
│   ├── A4 page rectangle 1 (white, border, shadow, 210×297mm at 96dpi ≈ 794×1123px)
│   │   └── 15mm interior padding (paged.js @page margin)
│   │       └── theme-{template} wrapper + TEMPLATE_STYLES[template].container
│   │           └── User-rendered HTML
│   │
│   ├── 16px gap
│   │
│   ├── A4 page rectangle 2 (only if content overflows)
│   │   └── …
│   │
│   └── 16px gap (between further pages)
│
└── Floating "Page X of N" pill (position: sticky / absolute, see below)
```

### Page counter placement

- **Position:** absolute / sticky inside the preview-pane viewport, anchored to bottom-right with 16px offset from both edges (`bottom-4 right-4` in Tailwind).
- **z-index:** above page rectangles, below modals (none exist this phase).
- **Visibility on scroll:** stays in viewport corner as user scrolls through stacked pages (sticky behavior).
- **Mobile:** same bottom-right placement inside the preview tab pane. Visible only when the Preview tab is active (the existing `MobileTabs.tsx` toggles which pane is mounted/visible).

### Page counter live-update behavior

- The counter updates synchronously when paged.js's Previewer emits its `rendered` (or equivalent) event with the new `counter(pages)` value.
- It is driven by the **existing 150ms `debounceRef` in `App.tsx`** (CONTEXT.md D-02) — no parallel debounce.
- During reflow (between when the user types and when paged.js finishes): the **previous count remains displayed** (no flicker, no "..." spinner). When the new count arrives, it swaps in place.
- Source of truth: paged.js native page count (CONTEXT.md D-03), not DOM node count.

### Loading / first-paint state

- Before paged.js has run for the first time (first paint, before its async render completes): show the gray pane background with NO page rectangle and the placeholder counter `Page – of –` in the bottom-right. No skeleton, no spinner.
- This state typically lasts < 150ms on a modern laptop. Adding a spinner would create visual noise for a state most users won't perceive.

### Empty state (no markdown content)

- When `htmlContent.trim()` is empty: skip paged.js entirely. Render the existing centered placeholder text (`Start typing markdown to see your resume preview`) on the gray (`bg-gray-100`) surround. No page rectangle. No counter (because there is nothing to count).
- This preserves the existing empty-state contract from `Preview.tsx`.

### Responsive behavior at narrow preview-pane widths

- A4 at 96dpi is ~794px wide. When the preview pane is narrower than `794px + 32px gutter = 826px` total, the page rectangle exceeds the pane width.
- **Phase 7 behavior:** preview pane scrolls **horizontally** (in addition to vertically). The page rectangle keeps its true 210mm width. Scrollbar appears at the pane bottom.
- Cursor: default (no special drag affordance for the horizontal scroll — native browser scroll).
- This is an explicit placeholder. **Phase 9 (ZOOM-01) introduces auto-fit-to-pane-width scaling** to fix this. Do NOT pre-build zoom controls or scaling in Phase 7.
- On mobile (viewport < 768px), the Preview tab pane is full-viewport-width. Most modern phones (375–430px CSS px) will trigger horizontal scroll. This is intentional and acceptable for Phase 7; Phase 9 resolves it.

### Theme-class scoping (CRITICAL)

- The `.theme-${template}` class **MUST** wrap the content paged.js emits, so `theme-{classic,modern,minimal}.css` `@apply` rules still cascade onto user content.
- Concretely: paged.js's emitted `.pagedjs_page > .pagedjs_page_content > …` tree must have `.theme-${template}` as an ancestor — either applied to the outer paged.js mount root OR injected as a wrapper around the source HTML that paged.js consumes.
- `TEMPLATE_STYLES[template].container` (font/leading wrapper) applies to the **in-page content**, NOT to the outer page rectangle. The page rectangle itself has only structural/chrome styles (white background, border, shadow).
- The page rectangle (the white sheet) gets its visual treatment from the phase-7 stylesheet ONLY. Theme CSS does not style page chrome.

### Print / PDF path (UNCHANGED)

- The existing `#print-area` div + second `<Preview>` instance + `@page { size: A4 portrait; margin: 15mm }` in `src/index.css` **remain untouched** this phase.
- `handleExportPdf` in `App.tsx` continues to call `window.print()`.
- **Preview page chrome is screen-only.** When `@media print` activates, `#app-shell` hides (existing rule), `#print-area` shows the un-paginated `<Preview>`, and the browser print engine paginates from `@page`. The on-screen paged.js page rectangles are inside `#app-shell` and therefore hidden during print.
- Phase 10 unifies preview and print into a single paged.js path. Phase 7 must not regress today's PDF behavior.

### Reflow trigger

- Single trigger: `htmlContent` changes (already driven by the 150ms `debounceRef` in `App.tsx`) OR `selectedTemplate` changes (template switch should reflow because per-template font metrics affect how content paginates).
- No additional debounce. No abort-on-rapid-edits (the existing debounce already coalesces).
- Pane resize from the split-pane drag does NOT trigger reflow in Phase 7 (page width stays at 210mm; only horizontal scroll changes). Phase 9 will revisit this when zoom is introduced.

### Failure behavior

- If paged.js fails to render (script load failure, throws during pagination): the preview pane should display the existing un-paginated `<Preview>` output as a fallback (essentially today's behavior). No error toast, no error UI — this is a personal tool and silent fallback is acceptable.
- Implementation note for planner: wrap the paged.js render in a try/catch; on catch, fall back to the existing `dangerouslySetInnerHTML` path. Log the error to `console.error` for developer visibility.

## Visual Style Tokens (new CSS for this phase)

These go in `src/index.css` (or a sibling `src/styles/pages.css` imported from `index.css`):

| Token / class | Value | Purpose |
|---------------|-------|---------|
| Pane background | `background-color: #f3f4f6;` (Tailwind `bg-gray-100`) | Preview pane surround |
| Page rectangle box-shadow | `box-shadow: 0 2px 8px rgba(0, 0, 0, 0.08), 0 1px 2px rgba(0, 0, 0, 0.06);` | Subtle Word/Docs elevation |
| Page rectangle border | `border: 1px solid #e5e7eb;` | Crisp sheet edge |
| Page rectangle background | `background-color: #ffffff;` | A4 sheet |
| Page rectangle margin between siblings | `margin-bottom: 16px;` (last child no margin) | Gap between stacked pages |
| Counter pill | `background-color: rgba(17, 24, 39, 0.85); color: #fff; padding: 4px 8px; border-radius: 6px; font-size: 12px; line-height: 1.2; font-weight: 500;` | Floating page counter |
| Counter pill position | `position: sticky; bottom: 16px; right: 16px;` (or absolute within scroll viewport — implementation detail) | Bottom-right anchor |

Border-radius on page rectangles: **0** (real paper has square corners). This is intentional, not an oversight.
