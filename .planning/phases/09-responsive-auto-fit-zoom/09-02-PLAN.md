  <how-to-verify>
    Start the dev server: `npm run dev`, then open the URL printed (typically http://localhost:5173).

    **Test 1 — Wide pane shows 1:1 (success criterion 2):**
    1. Drag the split-pane separator all the way to the LEFT so the preview pane is as wide as possible.
    2. Open DevTools → Elements panel.
    3. Find the div that wraps `previewerRootRef` (it has class `pagedjs-scale-wrapper`).
    4. Confirm: it has NO `style` attribute at all (or the style attribute is empty / absent).
    5. The A4 page should appear at its natural size — pull out a ruler if you want; 210mm is roughly 79.4mm × CSS-px-density.
    6. EXPECTED: No `transform` in the DOM, page is at 1:1, no horizontal scroll, no shrink.

    **Test 2 — Drag from wide to narrow shows smooth scale-down (success criterion 1 + 3):**
    1. Slowly drag the split-pane separator to the RIGHT, narrowing the preview pane.
    2. Watch the A4 page rectangle.
    3. EXPECTED: As soon as the pane gets narrower than the A4 page width, the page starts visibly shrinking. The shrink is continuous (not snappy), the page stays horizontally centered, and the bottom of the gray pane region is just below the bottom of the shrunken page (no large empty gray void).
    4. EXPECTED: At NO point during the drag does a horizontal scrollbar appear at the bottom of the preview pane.
    5. EXPECTED: The "Page X of N" pill in the bottom-right stays the same size and remains pinned to the bottom-right corner. It does NOT shrink with the page.

    **Test 3 — DevTools console clean (D-03 loop guard):**
    1. Open DevTools → Console panel.
    2. Drag the split-pane separator vigorously back and forth multiple times.
    3. EXPECTED: No "ResizeObserver loop limit exceeded" warnings appear. No red errors related to scale or transform.

    **Test 4 — Page count is stable across scales (D-08 — no paged.js retrigger):**
    1. Type or paste enough content into the markdown editor to produce 2 pages. Note the "Page X of N" pill — should say "Page 2 of 2".
    2. Drag the split-pane separator narrow, then wide, then narrow again.
    3. EXPECTED: The pill continues to say "Page 2 of 2" the whole time. The page count must not flicker, change, or reset.
    4. EXPECTED: There is no visible re-render flash of the preview content during drag (paged.js does NOT re-run).

    **Test 5 — Mobile viewport (D-07 — same logic on mobile, success criterion 1):**
    1. Open DevTools → Toggle Device Toolbar (Cmd+Shift+M on Mac, Ctrl+Shift+M on Windows/Linux).
    2. Set viewport to a typical mobile size — iPhone SE (375 × 667) or Pixel 5 (393 × 851).
    3. Reload the page if needed. Tap the "Preview" tab to switch to the preview view.
    4. EXPECTED: The A4 page is visible, shrunk to fit the narrow viewport. NO horizontal scrollbar at the bottom of the preview area.
    5. EXPECTED: The full width of the A4 page (left margin to right margin) is visible — nothing cut off.

    **Test 6 — Print path unaffected (regression check):**
    1. Exit DevTools device toolbar; restore desktop layout.
    2. Click the "Export PDF" button.
    3. EXPECTED: PDF download starts, opens correctly. The PDF shows the resume at 1:1 (NOT shrunk). PDF page count matches the preview "Page X of N" pill. No visible artifacts of the scale transform in the PDF.

    **Test 7 — Empty editor (regression check):**
    1. Clear the markdown editor entirely.
    2. EXPECTED: The preview shows the "Start typing markdown to see your resume preview" empty-state message. No errors, no broken layout.

    If ALL of Tests 1–7 pass, type `approved` to resume.
    If ANY test fails, describe which test failed and what you saw instead.
