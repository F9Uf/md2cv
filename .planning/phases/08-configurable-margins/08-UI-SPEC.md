# Phase 8 тАФ UI Design Contract

> Visual and interaction contract for the configurable margin control strip: four numeric inputs (top / bottom / left / right) rendered in a second toolbar row below the header. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase introduces exactly one new interactive widget set тАФ four `<input type="number">` fields in a compact strip. All other chrome (header, split-pane, mobile tabs, page rectangles) is unchanged from prior phases.

## Layout & Interaction Specifics

### Strip placement in app shell

```
<div class="h-screen bg-gray-50 flex flex-col overflow-hidden" id="app-shell">
  тФВ
  тФЬтФАтФА <Header>              h-12  bg-gray-900   (UNCHANGED from Phase 7)
  тФВ
  тФЬтФАтФА <MarginControls>      h-9   bg-gray-900 border-b border-gray-700  (NEW тАФ Phase 8)
  тФВ
  тФФтФАтФА <main class="flex-1 flex min-h-0">
        тФЬтФАтФА (desktop) <SplitPane>
        тФФтФАтФА (mobile)  <MobileTabs>
```

The new `<MarginControls>` strip is inserted between `<Header>` and `<main>` in `App.tsx`. It is always visible тАФ not collapsible, not conditionally rendered.

### Strip internal layout

```
[  Top [___] mm    Bottom [___] mm    Left [___] mm    Right [___] mm    [Reset]  ]
   тЖС label above    тЖС label above       тЖС label above    тЖС label above    тЖС optional
```

Each margin unit is a `<div class="flex flex-col items-center gap-1">`:
- Row 1: `<label>` (`text-xs font-medium text-gray-300`)
- Row 2: `<div class="flex items-center gap-1">` containing `<input>` + `<span class="mm label">`

All four units are in a horizontal flex row with `gap-2` (`8px`) between units.

The whole strip uses `flex items-center justify-between px-4` (left-aligned inputs group, right-aligned Reset button if shown).

**Reset button (Claude's discretion тАФ include):** A single `Reset` button sits at the far right of the strip. It restores all four inputs to `15` and persists the reset to localStorage. Rationale: the context deferred this but did not exclude it; a reset that matches the existing button treatment adds discoverability at zero visual cost in the dark strip. It follows the same height and style as other `Header.tsx` tool buttons.

Reset button dimensions: `h-7 px-2` (28px height, compact тАФ smaller than the h-8 header buttons to stay within the 36px strip).

### Input spec

```
<input
  type="number"
  min="0"
  max="50"
  step="1"
  value={margin.top}         // controlled
  onChange={handler}
  className="h-7 w-14 px-1.5 rounded bg-gray-700 text-white text-[13px] border border-gray-600 focus:outline-none focus:ring-1 focus:ring-blue-400 text-center"
  aria-label="Top margin in millimetres"
/>
```

Key constraints (from CONTEXT.md D-02, D-03):
- `type="number"`, `step=1`, `min=0`, `max=50`
- No sliders
- Four independent inputs тАФ no linked symmetric mode
- Height `h-7` (28px) тАФ fits inside 36px strip with 4px vertical padding on each side
- Text centered (`text-center`) for numeric readability

### Update trigger & debounce

- **onChange fires on every keystroke** (CONTEXT.md D-04). No `onBlur`-only update.
- Margin state change feeds into `App.tsx`'s existing `htmlContent`/`selectedTemplate` dependency chain that triggers paged.js reflow. The existing 150ms `debounceRef` coalesces rapid keystrokes before reflow runs. No second debounce is added.
- Live preview reflow: user sees the page margins adjust within ~150ms of the final keystroke in a burst.

### Persistence (from CONTEXT.md D-06)

- localStorage key: `md2cv-margins`
- Shape: `{ top: number, right: number, bottom: number, left: number }` (values in mm as integers)
- Validation on load: each value must be a number in `[0, 50]`; any invalid/missing value falls back to `15` for that side
- Persist on every margin change (inside the same try/catch pattern as other `md2cv-*` keys)

### Default values

All four margins: `15` mm (from Phase 7 D-04 / CONTEXT.md D-01).

### Mobile behavior

- The margin strip is a full-width row above `<main>`, always visible regardless of which tab (Editor or Preview) is active in `MobileTabs`.
- On narrow viewports, the strip may need to scroll horizontally if all four inputs + labels exceed viewport width. Use `overflow-x-auto` on the strip's inner container. Alternatively, reduce the `gap` to `gap-1` (4px) at narrow widths тАФ implementation detail for executor.
- Touch target: the `h-7` (28px) input height is below the 44px minimum touch target guideline. Mitigate by wrapping the label+input unit in a `<label>` that covers the full click/tap area, giving the effective tap target the full strip height.

### Failure / edge cases

- **Value cleared by user (empty input):** Treat empty string as `NaN`. Do NOT reflow with `NaN` margin тАФ guard in the handler: `if (!isNaN(parsed)) setMargin(...)`. Input shows empty transiently; last valid state persists to localStorage.
- **Value outside [0, 50]:** Browser `min`/`max` attributes prevent form submission but not programmatic `onChange`. Guard in handler: clamp `Math.min(50, Math.max(0, parsed))` before updating state.
- **localStorage unavailable:** Catch the write silently (matching existing pattern). Default values are used on next load.
