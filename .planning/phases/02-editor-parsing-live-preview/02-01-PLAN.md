**Step 2: Create `src/types/resume.ts`** with these exact type definitions:

```typescript
export interface ResumeEntry {
  title: string        // from h3
  details: string[]    // from bullet list items under the h3
  extra: string        // any non-bullet content directly under h3 (paragraphs, inline markup), rendered as raw HTML
}

export interface ResumeSection {
  heading: string      // from h2
  entries: ResumeEntry[]
  extra: string        // any content directly under h2 that is not inside an h3 block, rendered as raw HTML
}

export interface ResumeData {
  name: string         // from h1 (first h1 only; empty string if none)
  sections: ResumeSection[]
  preamble: string     // any content between h1 and the first h2, rendered as raw HTML (e.g., contact info paragraph)
}
```

**Step 3: Create `src/lib/parseResume.ts`.**

Per D-04: Use markdown-it to parse the markdown string into tokens, then walk the token array to build a `ResumeData` object.

Implementation approach:
- `import MarkdownIt from 'markdown-it'`
- Create a singleton `const md = new MarkdownIt()` at module scope
- Export `function parseResume(markdownText: string): ResumeData`
- Call `md.parse(markdownText, {})` to get the token array
- Walk tokens sequentially:
  - `heading_open` with `tag === 'h1'` -> next `inline` token's `content` becomes `name` (first h1 only; subsequent h1s treated as regular content)
  - `heading_open` with `tag === 'h2'` -> start a new `ResumeSection` with `heading` from the next `inline` token's `content`
  - `heading_open` with `tag === 'h3'` -> start a new `ResumeEntry` within the current section, `title` from next `inline` token
  - `bullet_list_open` when inside an h3 entry -> collect `list_item` children's inline content into `details[]`
  - All other tokens (paragraphs, blockquotes, hr, etc.) per D-05: render them using `md.renderer.render([token_group], md.options, {})` and append to the appropriate `extra` field (preamble if before first h2, section.extra if inside h2 but not h3, entry.extra if inside h3)
- Return the completed `ResumeData`

Per D-05: NO content is silently dropped. Unrecognized markdown renders as raw HTML in the `extra` fields.

**Component interface:**

```typescript
interface EditorProps {
  value: string
  onChange: (value: string) => void
}

export default function Editor({ value, onChange }: EditorProps)
```

**Implementation details:**

1. Use a `useRef<HTMLDivElement>(null)` for the mount container and `useRef<EditorView>` for the EditorView instance.

2. In a `useEffect` (runs once on mount), create the EditorView:
   ```typescript
   import { EditorView, lineNumbers, drawSelection, highlightActiveLine } from '@codemirror/view'
   import { EditorState } from '@codemirror/state'
   import { markdown } from '@codemirror/lang-markdown'
   import { languages } from '@codemirror/language-data'
   import { defaultKeymap, history, historyKeymap } from '@codemirror/commands'
   import { keymap } from '@codemirror/view'
   ```

3. Extensions array:
   - `markdown({ codeLanguages: languages })` -- per D-01: markdown syntax highlighting
   - `EditorView.lineWrapping` -- per D-03: word wrap enabled
   - `lineNumbers()`
   - `highlightActiveLine()`
   - `drawSelection()`
   - `history()`
   - `keymap.of([...defaultKeymap, ...historyKeymap])`
   - `EditorView.updateListener.of((update) => { if (update.docChanged) { onChange(update.state.doc.toString()) } })`
   - `EditorView.theme({ '&': { height: '100%' }, '.cm-scroller': { overflow: 'auto' } })` -- fills parent container

4. Create state: `EditorState.create({ doc: value, extensions })`
5. Create view: `new EditorView({ state, parent: containerRef.current })`
6. Store in viewRef.current
7. Cleanup: `viewRef.current?.destroy()` in useEffect return

8. When `value` prop changes externally (not from user typing), update the editor:
   - Use a second `useEffect` watching `value` -- but skip if the change originated from the editor's own updateListener (use a ref flag `isInternalChange` set to `true` inside the updateListener callback, checked and reset in the external sync effect). This prevents cursor-jumping loops.
   - When external change detected: `viewRef.current.dispatch({ changes: { from: 0, to: viewRef.current.state.doc.length, insert: value } })`

9. Render: `<div ref={containerRef} className="h-full w-full" />`

The component is a controlled-ish component: it takes `value` and `onChange`, but manages its own internal state via CodeMirror. The `onChange` callback fires on every keystroke, enabling real-time preview updates per D-06 (debounce is handled by the consumer in Plan 02).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| markdown input -> parser | User-authored markdown is parsed by markdown-it into tokens |
| markdown-it -> HTML render | markdown-it renders tokens to HTML strings stored in `extra` fields |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing (S) | parseResume | accept | No auth in scope -- personal single-user tool |
| T-02-02 | Tampering (T) | markdown-it tokens | accept | Parser reads tokens in memory, no persistence boundary to tamper |
| T-02-03 | Information Disclosure (I) | Editor content | accept | Client-side only, no data leaves the browser |
| T-02-04 | Injection (XSS) | extra fields containing raw HTML from markdown-it | mitigate | markdown-it has HTML disabled by default (html option is false). The `extra` fields contain markdown-it rendered output which escapes HTML entities. Do NOT enable `html: true` on the MarkdownIt instance. Verify in parseResume.ts that `new MarkdownIt()` is called without `{ html: true }`. |
| T-02-05 | Elevation of Privilege (E) | CodeMirror | accept | Editor is input-only, no privileged operations |
</threat_model>
