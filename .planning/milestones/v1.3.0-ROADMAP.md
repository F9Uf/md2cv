# Milestone v1.3.0: Support preview with realistic page

**Status:** ✅ SHIPPED 2026-05-21
**Phases:** 7–10
**Total Plans:** 10

## Overview

Replaced the continuous flowing preview div with a paged.js-driven render path that shows the resume as one or more real A4 page rectangles with configurable margins, a live "Page X of N" indicator, and responsive auto-fit zoom. Unified the preview and PDF export pipelines so the exported PDF is produced from the same DOM the user sees — eliminating the `templateInlineStyles.ts` / `ExportTarget` parallel-map workaround and retiring `html2pdf.js` entirely.

## Phases

### Phase 7: Page Chrome & Auto Pagination

**Goal**: Preview pane shows the resume as one or more real A4 pages with visible margins and a live page count, replacing the continuous flowing div.
**Depends on**: Phase 6 (Tailwind-Powered Preview Rendering)
**Requirements**: PREV-01, PREV-02, PREV-03
**Plans**: 3 plans

Plans:
- [x] 07-01-PLAN.md — Install pagedjs dependency + minimal TS ambient declaration
- [x] 07-02-PLAN.md — Create src/styles/pages.css page chrome + import from main.tsx
- [x] 07-03-PLAN.md — Preview.tsx paged.js integration + Page X of N pill + App.tsx enablePagination wiring

**Details:**
- `pagedjs@0.4.3` installed as runtime dependency with a minimal ambient TS declaration (`src/types/pagedjs.d.ts`) covering only the `Previewer` / `PagedjsFlow` / `polisher` / `chunker` surface — no `any` blanket
- `src/styles/pages.css` styles paged.js's emitted `.pagedjs_page` as white A4 sheets on a gray surround with drop shadow + 16px stack gap, with an `@media print` chrome-strip
- `Preview.tsx` rewritten to instantiate a fresh `new Previewer()` per reflow, clear the mount manually, pass `pagedjs_inline` stylesheets (prevents destructive `removeStyles()` default), and destroy `polisher`/`chunker` on cleanup
- `enablePagination` prop added (default `true`) — `App.tsx` passes `false` initially for the `#print-area` sibling so paged.js isn't double-instantiated on every keystroke (later flipped to `true` in Phase 10)

### Phase 8: Configurable Margins

**Goal**: User can set and persist page margins (top / bottom / left / right) through a UI control and the preview reflects the values immediately.
**Depends on**: Phase 7
**Requirements**: MARG-01, MARG-02, MARG-03
**Plans**: 2 plans

Plans:
- [x] 08-01-PLAN.md — MarginControls component + App.tsx state wiring
- [x] 08-02-PLAN.md — Preview.tsx dynamic @page CSS + human verification

**Details:**
- New `MarginControls` component renders a four-input strip (Top / Bottom / Left / Right) with `type="number"`, `min=0 max=50 step=1`, onChange clamping, and a Reset button that restores 15mm on all four sides
- App-level `margins` state initialized from `localStorage["md2cv-margins"]` with per-side validation; `handleMarginChange` persists writes via try/catch
- `Preview.tsx` accepts an optional `margins?: MarginValues` prop and injects `@page { size: A4 portrait; margin: Tmm Rmm Bmm Lmm }` via the `pagedjs_inline` stylesheet; `margins` added to the paged.js `useEffect` dependency array to trigger reflow

### Phase 9: Responsive Auto-Fit Zoom

**Goal**: The A4 preview page is always fully visible within the preview pane, scaling down when the pane is narrower than the page so no horizontal scroll is needed.
**Depends on**: Phase 7
**Requirements**: ZOOM-01
**Plans**: 2 plans

Plans:
- [x] 09-01-PLAN.md — Preview.tsx ResizeObserver auto-fit scale + pages.css print reset
- [x] 09-02-PLAN.md — Human verification of auto-fit zoom behavior

**Details:**
- Initial implementation used `transform: scale()` + a clip container — caused desktop blink on resize and clipped pages 4+ on multi-page content
- Reworked to CSS `zoom` (which affects layout, not just paint) on a `.pagedjs-scale-wrapper` div, with a hardcoded A4 width (793.7px) to eliminate the measurement feedback loop
- Mobile (≤767px) uses a fixed `zoom: 0.5`; both paths gate zoom application behind `pageCount !== null` so paged.js measures the natural A4 size first
- `pages.css` `@media print` reset: `.pagedjs-scale-wrapper { zoom: 1 !important; transform: none !important; height: auto !important }` — load-bearing for mobile PDF parity in Phase 10

### Phase 10: Unified Pixel-Perfect PDF Pipeline

**Goal**: The exported PDF is produced from the same DOM the user sees, matching the preview 1:1 across page size, margins, page-break positions, fonts, and colors — eliminating the `templateInlineStyles.ts` parallel workaround.
**Depends on**: Phase 7, Phase 8, Phase 9
**Requirements**: PDFX-01, PDFX-02
**Plans**: 3 plans

Plans:
- [x] 10-01-PLAN.md — App.tsx print-mount paged.js flip + index.css @page refactor + pages.css invariant verification
- [x] 10-02-PLAN.md — Remove html2pdf.js dead deps + update README.md and CLAUDE.md tech-stack lines
- [x] 10-03-PLAN.md — Human verification of preview ↔ PDF parity across 3 templates, mobile, and margin propagation

**Details:**
- `#print-area` flipped from `enablePagination={false}` (plain HTML stub) to `enablePagination={true}` so it runs its own paged.js `Previewer`; `aria-hidden="true"` added to prevent AT double-announcement
- `src/index.css` rewritten: removed static `@page { margin: 15mm }` (contradicted dynamic `@page` injected by paged.js), added `@page { size: A4 portrait; margin: 0 }` inside `@media print` (paged.js's per-page margins already inside `.pagedjs_page`, so browser margin would double-gutter), and replaced screen-side `display: none` with off-screen positioning so paged.js can measure real layout
- 21 transitive packages removed from `package-lock.json` (html2pdf.js, html2canvas, jspdf, canvg, pako, fflate, raf, etc.); README.md and CLAUDE.md tech-stack lines now read "paged.js + browser print"
- **Two regressions caught at human UAT and fixed inline in commit `738bea4`:**
  - **Blank PDF pages from bleeding chrome:** `#print-area` reused the full `<Preview/>` (including `overflow: auto` scroll container + sticky pill), which becomes a fragmentation container in print. Fixed by extracting a minimal `<PrintMount/>` component with zero on-screen chrome.
  - **`left: -9999px` removed print-area from print flow:** Replaced with `position: fixed; visibility: hidden; pointer-events: none; z-index: -1` for the screen-side cloak, flipped to `position: static; visibility: visible` in `@media print`. Element stays in real layout (paged.js measurement works) and reaches the print engine cleanly.

## Milestone Summary

**Key Decisions:**

- **D-PDF-1 (retired html2pdf.js):** Client-side PDF via html2pdf.js (v1.0 decision) was retired in favor of paged.js + browser-native print over the same DOM the preview uses. The canvas-rasterization path forced a parallel hex-color map (`templateInlineStyles.ts`) to avoid oklch crash in html2canvas, and could never reach pixel parity with the preview. paged.js + print resolves both problems at once.
- **D-10-PRINTMOUNT (Phase 10):** Dedicated `<PrintMount/>` component instead of reusing `<Preview/>` for the off-screen print mount — minimal mount eliminates the bleeding-chrome bug class entirely.
- **D-10-CLOAK (Phase 10):** `position: fixed; visibility: hidden` cloak for `#print-area` instead of `left: -9999px` — element stays in real layout for paged.js measurement and is reachable by the print engine.
- **D-08-DEPSARRAY (Phase 8):** `margins` added to the paged.js `useEffect` dependency array — single source of truth for "reflow when margins change."
- **D-09-CSSZOOM (Phase 9):** CSS `zoom` (which affects layout) chosen over `transform: scale` (paint-only, breaks clip containers) for auto-fit scaling.
- **D-09-PAGECOUNTGATE (Phase 9):** Zoom application gated behind `pageCount !== null` — paged.js always measures natural A4 size first.

**Issues Resolved:**

- Continuous flowing preview replaced with real A4 page chrome and live page count
- Preview ↔ PDF parity achieved across all three templates, multi-page content, mobile viewport, and dynamic margins
- `templateInlineStyles.ts` parallel hex-color map workaround retired (and the file removed from src/)
- Supply-chain surface reduced (21 transitive packages dropped from lockfile alongside html2pdf.js)

**Issues Deferred / Tech Debt:**

- Phase 8 UI-SPEC cosmetic deviations on the margin strip (header is `bg-gray-700` vs spec `bg-gray-900`; labels UPPERCASE vs spec Title-case; Reset is SVG icon vs spec text button) — functional must-haves all SATISFIED, user accepted by shipping
- Phase 10 has no phase-level `10-VERIFICATION.md` — only plan-scoped `10-03-VERIFICATION.md` carries the sign-off
- `PrintMount` lacks a `hasError` fallback path (parity with `Preview.tsx`) — surfaced in Phase 10 code review (WR-01); a paged.js render failure produces a silent blank PDF rather than a visible error state
- Phase 7 frontmatter `status: complete` lags body `**Status:** human_needed` — body never reconciled after 07-HUMAN-UAT.md all-pass approval
- Bundle size: pagedjs adds ~1.3MB to the largest chunk (`index-*.js: 1,320 kB minified / 391 kB gzip`); acceptable for a personal tool, could be code-split via dynamic import if needed later
