# Milestone v1.2.0: Support render HTML with Tailwind classes

**Status:** ✅ SHIPPED 2026-04-26
**Phases:** 5–6
**Total Plans:** 5

## Overview

Simplified the markdown-to-preview pipeline by replacing the token-walking parser with `md.render()`, restructured template styles from semantic keys to HTML element tags, and enabled full Tailwind CSS runtime support in the preview so users can write arbitrary Tailwind utility classes in their markdown HTML.

## Phases

### Phase 5: Parser Simplification & Template Restructure

**Goal**: The markdown-to-HTML pipeline is simplified and template styles are restructured so that any HTML element can be styled by tag
**Depends on**: Phase 4 (shipped)
**Requirements**: PARSER-01, PARSER-02, TMPL-01, TMPL-02
**Plans**: 3 plans

Plans:
- [x] 05-01-PLAN.md — Rewrite parseResume.test.ts for HTML string output + snapshot baseline
- [x] 05-02-PLAN.md — Restructure templateStyles.ts to element-keyed format
- [x] 05-03-PLAN.md — Replace parseResume() body with md.render(), simplify Preview.tsx stub, update App.tsx

**Details:**
- Replaced the entire token-walking parser with `md.render()` — parseResume.ts went from ~120 lines to 5
- `TemplateClasses` interface restructured from semantic keys (name, sectionHeading, entryTitle…) to HTML element tags (h1, h2, h3, p, ul, ol, li, code, pre, a, blockquote, hr)
- `src/types/resume.ts` deleted — ResumeData/ResumeSection/ResumeEntry types fully eliminated
- Preview.tsx rewritten as a minimal HTML stub using `dangerouslySetInnerHTML`
- App.tsx state changed from `resumeData: ResumeData` to `htmlContent: string`
- All 15 tests turned GREEN after md.render() swap

### Phase 6: Tailwind-Powered Preview Rendering

**Goal**: The live preview applies template Tailwind classes to rendered HTML elements and user-authored Tailwind utility classes work at runtime
**Depends on**: Phase 5
**Requirements**: PREV-01, PREV-02, PREV-03
**Plans**: 2 plans

Plans:
- [x] 06-01-PLAN.md — Create themes.css (@apply rules for all 3 themes), simplify templateStyles.ts, add Play CDN to index.html
- [x] 06-02-PLAN.md — Update Preview.tsx to apply theme-{template} class + browser verification checkpoint

**Details:**
- CSS theme architecture using Tailwind v4 `@custom-variant` for scoped per-template element styling (deviated from plan's `@layer base` — valid alternative mechanism)
- Three per-theme CSS files: theme-classic.css, theme-modern.css, theme-minimal.css — imported via themes.css aggregator
- `templateStyles.ts` simplified to container strings only; element styles moved to CSS
- Tailwind Play CDN added to index.html for runtime utility class resolution
- DOMPurify configured with `ADD_ATTR: ['class']` so user-authored Tailwind classes survive sanitization
- `theme-${template}` class wired onto Preview container div to activate scoped CSS variants
- User tuned modern/minimal theme styles during browser verification (p/a colors, added `strong` rule)

## Milestone Summary

**Key Decisions:**
- `md.render()` over token-walking: simpler pipeline, HTML string as the single data format
- Tailwind v4 `@custom-variant` over `@layer base`: discovered during build — valid alternative that scopes styles to theme container class
- Play CDN for runtime Tailwind: enables arbitrary user-authored classes without build-time scanning
- DOMPurify `ADD_ATTR: ['class']`: required to let user-authored class attributes through sanitization

**Issues Resolved:**
- DOMPurify was stripping class attributes (PREV-03 blocker) — fixed by adding `ADD_ATTR: ['class']` to sanitize call
- `@apply` failing in separate CSS files under Tailwind v4 — fixed by adding `@reference "tailwindcss"` directive
- Root element guard needed in main.tsx — explicit null check added

**Technical Debt Incurred:**
- `@custom-variant` CSS architecture deviates from plan; documented in 06-VERIFICATION.md
- ExportTarget.tsx still uses parallel `templateInlineStyles.ts` (hex/rgb) — not updated for new element-keyed system; must be kept in sync manually if styles change significantly
