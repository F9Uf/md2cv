# Milestone v1.1.0: Support text styles & HTML

**Status:** ✅ SHIPPED 2026-04-24
**Phases:** 4
**Total Plans:** 2

## Overview

Enable inline markdown styles in bullet points and full HTML rendering throughout the resume markdown. Users can now write `**bold**`, `*italic*`, `` `code` ``, `[links](url)`, and arbitrary HTML in their resume markdown and see it rendered correctly in the live preview and PDF export.

## Phases

### Phase 4: Inline Styles & HTML Preview

**Goal**: Users can write rich inline markdown and HTML anywhere in their resume and see it rendered correctly in the preview
**Depends on**: Phase 3 (v1.0 complete)
**Requirements**: STYLE-01, STYLE-02, STYLE-03, STYLE-04, HTML-01, HTML-02
**Plans**: 2 plans

Plans:

- [x] 04-01-PLAN.md — Enable html:true in parser and render bullet details as inline HTML
- [x] 04-02-PLAN.md — Wire dangerouslySetInnerHTML in Preview and ExportTarget for detail items

**Details:**

Plan 01: Updated `src/lib/parseResume.ts` to instantiate MarkdownIt with `{ html: true }` and call `md.renderInline(t.content)` on bullet detail tokens, converting inline markdown to HTML strings. Updated `src/types/resume.ts` JSDoc to document that `details` values are HTML strings.

Plan 02: Updated `src/components/Preview.tsx` and `src/components/ExportTarget.tsx` to render bullet detail `<li>` elements using `dangerouslySetInnerHTML={{ __html: detail }}`, aligning with the existing pattern for `extra` fields.

## Milestone Summary

**Key Decisions:**

- D-10 updated: MarkdownIt now uses `html: true` — XSS risk accepted because this is a personal single-user tool (no other users)
- `md.renderInline()` chosen over `md.render()` for bullet details to avoid wrapping `<p>` tags
- `dangerouslySetInnerHTML` pattern consistent with existing `extra` field rendering in both Preview and ExportTarget

**Issues Resolved:**

- Raw markdown syntax (`**bold**`) was showing as literal text in preview bullet points — fixed by rendering inline HTML at parse time

**Issues Deferred:**

- STYLE-05: Inline styles in PDF export — Phase 5 was planned and reverted; accepted as-is since html2pdf.js renders the ExportTarget which already uses dangerouslySetInnerHTML
- HTML-03: HTML renders in PDF export — same as above
- Phase 04 verification: human_needed status — deferred to next milestone
- Dark mode for CodeMirror — quick task dropped

**Technical Debt Incurred:**

- ExportTarget's parallel `templateInlineStyles.ts` must be manually kept in sync with `templateStyles.ts` if template styles change significantly (pre-existing from v1.0 oklch workaround)
